SQL Tutoring Raport - Scholarly HTML

Introducere

Acest document contine o prezentare a apllicatiei web SQL-Tutoring, realizata de
Maxim Maria Valeria si Popa Ioana-Alexia in cadrul disciplinei Tehnologii Web.

Conventii

Acest document a fost redactat luand in consideratie prevederile oferite de macheta IEEE System Requirements Specification Template

Scopul produsului

Scopul aplicatiei este de a facilita invatarea limbajului SQL prin rezolvarea si crearea unor exercitii aferente domeniului.

Publicul tinta

Documentul este destinat persoanelor ce sunt in curs de invatare a limbajului SQL ce vor folosi aplicatia SQL-TWO, precum si eventualilor dezvoltatori software care vor dori extinderea functionalitatilor aplicatiei.

Caracteristicile aplicatiei

Functionalitatile utilizatorilor

Functionalitatile aplicatiei sunt in functie de rol.

Functionalitatile elevului

  1. Crearea unui cont, logare
  2. Modificarea datelor contului
  3. Rezolvarea unor exercitii si marcarea lor
  4. Adaugarea unor noi exercitii
  5. Vizualizarea exercitiilor rezolvate si adaugate
  6. Vizualizarea unor statistici despre exercitiile propuse

Functionalitatile administratorului:

  1. Logare
  2. Administrarea conturilor si a exercitiilor
  3. Vizualizarea unor statistici despre toate exercitiile
  4. Importarea si exportarea exercitiilor

Detalii de implementare

In cadrul dezvoltarii interfetei s-a tinut cont de unele bune practici. In primul rand, paginile sunt proiectate pentru a fi responsive. De asemenea, paginile sunt in acelasi stil, toate urmand un sablon bazat pe integrarea de sectiuni marcate.

Securitate

Pentru a fi procesate, cererile spre server trebuie sa contina in header token-ul de autorizare JWT, primit in urma logarii. Aditional, unele endpoint-uri verifica si rolul utilizatorului ce a facut cererea, pentru a permite doar unele operatiuni.

Arhitectura aplicatiei

Aplicatia este contruita pe baza arhitecturii SOA (Service Oriented Architecture). Astfel, clientul face request-uri catre server, iar acesta din urma raspunde strict cu datele necesare. In acest fel, clientul este responasbil de randarea paginilor, iar serverul de procesarea datelor.

Pentru a facilita dezvoltarea aplicatiei, avem o arhitectura bazata pe layers. Astfel, cererile, initial ajung in layer-ul de routare, care le trimite mai departe catre layer-ul de servicii, apoi sunt trimise catre layer-ul de repositories, care se ocupa de accesul la baza de date.

Accesul la baze date

Singurul layer responsabil de accesul la baze de date este repository-ul. Acesta, insa, foloseste un connection pool pentru a reutiliza conexiunile la baza de date, pentru a nu fi nevoie de a crea o noua conexiune pentru fiecare cerere, ceea ce ar reprezenta o operatie care ar afecta performanta.

Functionalitati

Autentificare

Pentru inregistrarea utilizatorul trebuie sa specifice: numele de utilizator, email-ul, parola, numele si prenumele. In cazul in care acestea sunt valide si utilizatorul si email-ul nu au mai fost utilizate, se va efectua inregistrarea, in caz contrar un mesaj corespunzator va fi afisat. In urma inregistrarii, parolei i este aplicat un hash, dupa care este salvata in baza de date.

Pentru logare, utilizatorul trebuie sa specifice numele de utilizator si parola. In cazul in care acestea sunt valide, serviciul web returneaza un JWT care este stocat in local storage si este utilizat pentru request-urile viitoare catre server.

Probleme

Problemele propuse studentului spre rezolvare cresc in dificultate pe masura ce rezolva mai multe probleme. Dupa ce a rezolvat un numar de probleme, studentul este obligat sa propuna o noua problema, dupa care poate rezolva in continuare. In momentul in care a rezolvat o problema, aceasta poate sa ii marcheze dificulatatea sau sa rezolve alta problema.

Daca studentul considera, el poate marca o problema ca fiind gresita. Acesta poate face acest lucru de 5 ori, dupa ce aceasta limita este depasita, studentul trebuie sa astepte ca admin-ul sa gestioneze problemele marcate drept gresite, pentru ca el sa poata marca din nou astfel.

Toti utilizatorii aplicatiei (studenti/adimin) pot vizualiza comentarii lasate pentru problemele propuse spre rezolvare si, de asemenea, pot posta propriul comentariu.

Administrare

Administratorul are acces la toate problemele si poate sa le stearga, dar si sa importe, atat din format xml/json, cat si sa exporte problemele in aceste formate. Pe langa aceasta, mai are acces la diverse statistici despre problemele rezolvate, dar si despre utilizatorii inregistrati.

Administratorul mai are posibilitatea de a gestiona problemele marcate drept gresite de catre elevi. Astfel, poate accepta o astfel de marcare, problema fiind ulterior stearsa din baza de date sau o poate respinge.
